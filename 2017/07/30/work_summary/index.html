工作总结

以下是我个人对于工作的总结，定期总结更新。。。

1、编码规范

养成一个好的编码规范，好的命名规则，是一个好的程序员的必须要走的路。
可以借鉴google编码规范

2、新的需求应该如何做

  • 首先站在用户的角度上分析需求的合理性，尽量简单化需求。（解决简单的问题还可以用简图，简单直观。）

  • 然后设计多个总体架构（做到敢想、敢试），推理成功后选择一个合适的（框架合理、模块独立）再下手编程。（注意：需求更改很正常，多方面考虑问题。为了提高进度，没有真图，用线框图。）

  • 逻辑放到纸上进行推理和修改，考虑长远，少打补丁，减少与其他模块的耦合度。

  • 任何一个交互程序，都要做到数据和程序代码分开，合理使用配置文件。（考虑MVC模式）

  • 着手写代码，如果感觉代码越写越复杂，要停下来重新设计并推理。

  • 完成开发后，写测试用例进行自我检测，要一点一点的检查，确定没有问题再去提交代码。

3、工作越到问题（bug）的解决方法

  • 不要为了解决问题而解决问题，而不去思考设计者的想法。一切都要以实际出发。

  • 跳出程序的思维，用其他方法解决这个问题。需求更改、其他工具、换一种方法去实现。
    例如：用正则表达式 + Excel去 解决 key - value 问题。

  • 不要急于解决问题，有官网找官网，如果没有就google, 学习使用如何用谷歌搜索

  • 找与自己问题相关的帖子。在demo上测试，成功了在移植到项目中，进行调试。

4、解决问题的经验

不要盲目的去尝试各种参数的改变会调试好东西，而是需要退一步，站在更高的地方的，换一种思路去解决问题。考虑自己的测试工具的问题，不同工具的差异化，而不是局限于配置产品本身的问题。考虑问题好多样化。

如：配置ubuntu的shadowsocks，服务器已经配置好了，但是客户端连接好后怎么也上不了外网，于是更改服务器的配置很多次，无效。又更改客户端的设置很多次，也无效。最后是火狐浏览器不支持，需要安装插件才可以使用代理。用safari就可以上外网，不需要设置任何东西。

5、工作处理的流程

任务和BUG都按照优先级高低进行。

6、工作态度

态度要严谨，特别是与他人有交互的地方。

培养思维模式，不要受到语言、平台的局限性。

7、程序员的容易出现的问题

  • 解决问题的方法越来越复杂，应该跳出看看是否走错路了。

  • 问题看的不透彻，对于源码的研究不够透彻，对于问题的分析能力不够全面、具体。

8、如果复杂的代码，可以添加测试用例

如：

subscribe to my blog by scanning my public wechat account
坚持原创技术分享，您的支持将鼓励我继续创作！
